# Opik 中文本地化计划

## 概述

将 Opik 平台完整本地化为中文版本，包括：
1. **前端国际化**：引入 i18n 框架，提取并翻译所有 UI 文本
2. **文档翻译**：翻译 v2 版本的 248 个 mdx 文档
3. **代码注释**：翻译和添加中文注释

## 项目规模

| 部分 | 规模 | 说明 |
|------|------|------|
| 前端 | 2431 个 TS/TSX 文件 | 约 1300+ 个硬编码字符串需要翻译 |
| 文档 | 566 个 mdx 文件 | v2: 248 个, v1: 318 个（优先 v2） |
| 代码注释 | Java + Python + TypeScript | 需要翻译和添加中文注释 |

## 一、前端国际化

### 1.1 技术选型

**框架**：react-i18next + i18next

**依赖**：
```bash
npm install react-i18next i18next i18next-browser-languagedetector i18next-http-backend
```

**优势**：
- React 生态最成熟的 i18n 方案
- 原生 TypeScript 支持
- 命名空间分离，适合大型项目
- 懒加载语言包，不影响首屏性能

### 1.2 文件结构

```
apps/opik-frontend/src/
├── i18n/
│   ├── index.ts                    # i18n 初始化配置
│   ├── config.ts                   # 语言配置常量
│   └── locales/
│       ├── en/                     # 英文语言包
│       │   ├── common.json         # 通用文本（按钮、标签等）
│       │   ├── navigation.json     # 导航菜单
│       │   ├── actions.json        # 操作文本
│       │   ├── forms.json          # 表单文本
│       │   ├── messages.json       # 提示消息
│       │   ├── dialogs.json        # 对话框文本
│       │   └── pages/              # 页面专用文本
│       │       ├── projects.json
│       │       ├── datasets.json
│       │       ├── experiments.json
│       │       └── ...
│       └── zh/                     # 中文语言包（结构同上）
│           ├── common.json
│           ├── navigation.json
│           └── ...
```

**任务**：

4. 创建语言切换组件

#### Phase 2：核心页面翻译（3-5 天）

**优先级 P0**（用户直接可见的核心功能）：

| 区域 | 关键文件 | 字符串数 |
|------|---------|---------|

#### Phase 4：翻译完善（持续）

- 错误消息翻译
- Toast 消息翻译

- 动态内容翻译

### 1.4 翻译提取策略

**手动提取**（推荐用于初始版本）：
1. 遍历每个页面组件
2. 识别硬编码字符串
3. 替换为 `t('key')` 调用
4. 将字符串添加到对应的语言包

**自动提取辅助**：
```bash
# 使用 i18next-parser 自动提取
npx i18next-parser --config i18next-parser.config.js
```

### 1.5 语言切换组件

**位置**：`v2/layout/Header/LanguageSwitcher.tsx`

**功能**：
- 下拉菜单选择语言
- 持久化用户选择到 localStorage
- 支持中英文切换

## 三、代码注释翻译

### 3.1 范围

| 部分 | 文件数 | 优先级 |
|------|--------|--------|
| Python SDK | ~100 个 | P1 |
| TypeScript SDK | ~80 个 | P1 |
| Java 后端 | ~200 个 | P2 |

### 3.2 策略

**翻译对象**：
- 类/模块级注释
- 公开 API 的方法注释
- 复杂逻辑的解释性注释

**不翻译对象**：
- 变量名、函数名
- 代码内部注释（实现细节）
- 技术术语（如 API、SDK、HTTP 等）

### 3.3 实施方式

**Python SDK**：
- 使用 Google 风格 docstring
- 添加中文翻译行

```python
def evaluate(
    dataset: Dataset,
    task: Task,
    scoring_metrics: List[BaseMetric],
) -> EvaluationResult:
    """
    Evaluate a dataset using the provided task and metrics.
    
    评估数据集，使用提供的任务和指标。
    
    Args:
        dataset: The dataset to evaluate.
            要评估的数据集。
        task: The task to run on each dataset item.
            在每个数据集项上运行的任务。
        scoring_metrics: List of metrics to compute.
            要计算的指标列表。
    
    Returns:
        EvaluationResult containing scores and details.
        包含分数和详细信息的评估结果。
    """
```

**TypeScript SDK**：
- 使用 JSDoc 风格
- 添加中文翻译行

```typescript
/**
 * Evaluate a dataset using the provided task and metrics.
 * 评估数据集，使用提供的任务和指标。
 * 
 * @param dataset - The dataset to evaluate.
 *   要评估的数据集。
 * @param task - The task to run on each dataset item.
 *   在每个数据集项上运行的任务。
 * @returns Promise resolving to evaluation results.
 *   解析为评估结果的 Promise。
 */
```

**Java 后端**：
- 使用 Javadoc 风格
- 添加中文翻译行


## 八、风险和注意事项

### 8.1 技术风险

1. **翻译溢出**：中文文本可能比英文长，导致布局问题
   - 解决方案：使用 CSS `text-overflow: ellipsis` 和响应式设计

2. **动态内容**：包含变量的字符串需要特殊处理
   - 解决方案：使用 i18next 插值语法 `t('key', { count: 5 })`

3. **复数形式**：中英文复数规则不同
   - 解决方案：使用 i18next 复数功能

### 8.2 维护风险

1. **翻译同步**：新增功能需要同步更新翻译
   - 解决方案：在 CI/CD 中添加翻译完整性检查

2. **术语一致性**：不同翻译者可能使用不同术语
   - 解决方案：维护统一的术语表

### 8.3 性能风险

1. **语言包大小**：大量翻译可能增加包体积
   - 解决方案：使用命名空间分离，按需加载
