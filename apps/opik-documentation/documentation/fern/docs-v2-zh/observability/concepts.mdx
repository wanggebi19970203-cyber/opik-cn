追踪是 Opik 中可观测性的基础。它允许你通过捕获应用执行的详细信息来监控、调试和优化 LLM 应用。理解这些核心概念对于有效使用 Opik 的追踪功能至关重要。

## 概述

使用 LLM 应用时，了解底层发生了什么对于调试问题、优化性能和确保可靠性至关重要。Opik 的追踪系统通过在多个层级捕获详细的执行信息来提供全面的可观测性。

为了有效使用 Opik 的追踪功能，理解以下关键概念很重要：

1. **追踪（Trace）**：代表与 LLM 或 Agent 的单次交互的完整执行路径
2. **Span**：追踪中的单个操作或步骤，代表特定的动作或计算
3. **线程（Thread）**：形成连贯对话或工作流的相关追踪集合
4. **指标（Metric）**：对 AI 模型性能提供客观评估的量化测量
5. **优化（Optimization）**：系统化地改进和评估 LLM 提示词和配置的过程
6. **评估（Evaluation）**：使用各种指标系统化测试提示词和模型性能的框架

## 追踪（Trace）

**追踪**代表与 LLM 或 Agent 的单次交互的完整执行路径。可以将其视为一次请求-响应周期中发生的所有事情的详细记录。每个追踪捕获交互的完整上下文，包括输入、输出、时间以及任何中间步骤。

### 追踪的关键特征：

- **唯一标识**：每个追踪都有一个唯一标识符，允许你跟踪和引用它
- **完整上下文**：包含理解交互过程中发生什么所需的所有信息
- **时间信息**：记录交互何时开始、结束以及每个部分耗时多久
- **输入/输出数据**：捕获发送给 LLM 的确切提示词和收到的响应
- **元数据**：包括额外的上下文，如使用的模型、温度设置和自定义标签

### 示例用例：

- **调试**：当 LLM 产生意外输出时，你可以检查追踪以了解出了什么问题
- **性能分析**：通过分析追踪时间来识别瓶颈和慢操作
- **成本跟踪**：监控每次交互的 Token 使用量和相关成本
- **质量保证**：审查追踪以确保应用按预期运行

## Span

**Span**代表追踪中的单个操作或步骤。虽然追踪展示完整画面，但 Span 将执行分解为细粒度、可测量的组件。这种层级结构允许你同时理解 LLM 应用中的高层流程和详细操作。

### Span 的关键特征：

- **层级结构**：Span 可以包含其他 Span，在追踪内创建树状结构
- **特定操作**：每个 Span 代表一个独立的动作，如函数调用、API 请求或数据处理步骤
- **详细计时**：每个操作的精确开始和结束时间
- **上下文保持**：维护父子操作之间的关系
- **自定义属性**：可以包含特定于操作的额外元数据

### 常见 Span 类型：

- **LLM 调用**：对语言模型的单个请求
- **函数调用**：Agent 内的工具或函数调用
- **数据处理**：数据的转换或操作
- **外部 API 调用**：对第三方服务的请求
- **自定义操作**：你想要跟踪的任何用户定义操作

### 示例 Span 层级：

```
Trace: "Customer Support Chat"
├── Span: "Parse User Intent"
├── Span: "Query Knowledge Base"
│   ├── Span: "Search Vector Database"
│   └── Span: "Rank Results"
├── Span: "Generate Response"
│   ├── Span: "LLM Call: GPT-4"
│   └── Span: "Post-process Response"
└── Span: "Log Interaction"
```

## 线程（Thread）

**线程**是形成连贯对话或工作流的相关追踪集合。线程对于理解多轮交互和在多次 LLM 调用之间维护上下文至关重要。它们提供了一种将相关追踪分组的方式，使分析对话模式和用户旅程更加容易。

### 线程的关键特征：

- **对话上下文**：维护多轮交互的流程
- **追踪分组**：在单个线程标识符下组织相关追踪
- **时间排序**：线程内的追踪按时间顺序排列
- **共享上下文**：允许你查看上下文在整个对话中如何演变
- **跨追踪分析**：支持分析多个相关交互之间的模式

### 何时使用线程：

- **聊天应用**：将对话中的所有消息分组
- **多步工作流**：跟踪跨越多次 LLM 调用的复杂流程
- **用户会话**：组织来自单个用户会话的所有交互
- **Agent 对话**：跟踪 Agent 与用户之间的完整交互

### 线程管理：

线程通过定义 `thread_id` 并在追踪中引用来创建。这允许你：

- **维护上下文**：跟踪对话历史和用户状态
- **调试对话**：理解对话如何随时间演变
- **分析模式**：识别常见的对话流程和用户行为
- **优化性能**：发现多轮交互中的瓶颈

## 指标（Metric）

**指标**对 AI 模型的输出提供量化评估，支持客观比较和随时间跟踪性能。它们对于理解 LLM 应用的运行情况和识别改进领域至关重要。

### 指标的关键特征：

- **量化测量**：提供可以比较和跟踪的数值分数
- **客观评估**：消除性能评估中的主观偏差
- **趋势分析**：支持跟踪性能随时间的变化
- **比较分析**：允许在不同模型、提示词或配置之间进行比较
- **自动评估**：可以在无人干预的情况下自动计算

### 常见指标类型：

- **准确性指标**：衡量模型产生正确输出的频率
- **质量指标**：评估生成文本的质量（如连贯性、相关性）
- **效率指标**：跟踪延迟和吞吐量等性能特征
- **成本指标**：监控 Token 使用量和相关成本
- **自定义指标**：针对你的用例量身定制的领域特定测量

## 优化（Optimization）

**优化**是系统化地改进和评估 LLM 提示词和配置以提升性能的过程。它涉及迭代测试不同方法并使用数据驱动的洞察进行改进。

### 优化的关键方面：

- **提示词工程**：改进给 LLM 的指令
- **参数调优**：调整模型设置，如温度、top-p 和最大 Token 数
- **少样本学习**：优化上下文学习的示例选择
- **工具集成**：改进 LLM 与外部工具和函数的交互方式
- **性能监控**：跟踪改进和退化随时间的变化

## 评估（Evaluation）

**评估**提供了一个框架，用于使用各种指标系统化地测试提示词和模型在数据集上的性能。它是为 LLM 应用做出数据驱动决策的基础。

### 评估的关键组成部分：

- **数据集**：带有输入和预期输出的测试用例集合
- **实验**：测试特定配置的单次评估运行
- **指标**：性能的量化测量
- **比较分析**：不同方法的并排比较
- **统计显著性**：确保结果可靠且可复现

## 追踪最佳实践

<Steps>
  <Step title="1. 从清晰的追踪边界开始">
    定义什么构成单个追踪的清晰边界。通常，这应该与完整的用户交互或业务操作保持一致。
  </Step>
  <Step title="2. 使用有意义的 Span 名称">
    为 Span 选择描述性名称，清楚表明正在执行什么操作。这使调试更加容易。
  </Step>
  <Step title="3. 为对话使用线程 ID">
    为相关交互使用一致的线程 ID。这对于聊天应用和多步工作流尤为重要。
  </Step>
  <Step title="4. 添加相关元数据">
    包含对分析有用的自定义属性和元数据。考虑添加用户 ID、会话信息和业务上下文。
  </Step>
  <Step title="5. 持续监控性能">
    设置告警和仪表板来监控追踪性能、错误率和成本。这有助于你尽早发现问题。
  </Step>
  <Step title="6. 利用追踪进行优化">
    定期分析追踪以识别优化机会，如降低延迟或提高提示词有效性。
  </Step>
</Steps>

<Tip>
  **专业建议**：从基本追踪开始，随着你识别出需要更深入可观测性的区域，逐步添加更详细的 Span。不要试图一次性追踪所有内容 - 首先关注最关键的路径。
</Tip>

<Warning>
  **重要**：追踪时注意敏感数据。避免在追踪中记录个人身份信息（PII）或敏感业务数据。使用 Opik 的数据过滤功能来保护敏感信息。
</Warning>
