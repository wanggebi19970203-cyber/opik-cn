Opik 专为大规模关键工作负载而构建。无论您运行的是小型概念验证还是大规模企业部署，Opik 都能无缝适应您的需求。其无状态架构和强大的 ClickHouse 存储使其具有高度弹性、可水平扩展，并能应对未来的数据增长。

本指南概述了在生产环境中运行 Opik 的推荐配置和最佳实践。

## 参考部署：生产环境实例

以下基于一个活跃的大规模 Opik 部署，服务于真实生产用户。在规划您自己的基础设施时，可将其作为参考。

### 工作负载概况

| 指标 | 数值 |
|--------|-------|
| 活跃用户（每日） | ~600 |
| 每日摄入的 Trace 数量 | 400-600 万 |
| 每周数据摄入量 | ~100 GB |
| 存储的 Trace 总量 | 4000 万（400 GB） |
| 存储的 Span 总量 | 2.5 亿（3.1 TB） |
| 磁盘总数据量 | 5 TB |
| 每秒查询次数 | ~80 |
| 每秒插入次数 | ~20 |
| 每分钟插入行数 | 最高 75,000 |

### 基础设施

**Opik 后端** — 10 个 Pod
- 每个 Pod 3 个 CPU 核心，5 GB 内存

**Opik Python 后端** — 12 个 Pod
- 每个 Pod 1.5 个 CPU 核心，2 GB 内存

**Opik 前端** — 3 个 Pod
- 轻量级反向代理，CPU 和内存需求极低

**ClickHouse** — 2 个副本，1 个分片
- 每个副本 30 个 CPU 核心，230 GB 内存
- 每个副本 15 TB gp3 SSD 存储（3000 IOPS，250 MiB/s 吞吐量）

### 关键配置

| 配置项 | 数值 |
|---------|-------|
| 异步 ClickHouse 插入 | 已启用 |
| 速率限制 | 每个客户端每 60 秒 10,000 个事件 |
| 最大查询执行时间 | 60 秒 |
| 每次查询最大内存 | 10 GB |
| 摄入方式 | 批量端点（每请求最多 1,000 项） |

该部署在 ClickHouse 上的平均 CPU 利用率稳定在 10-20%，并有 40-50% 的余量应对流量高峰。

## 为增长而设计

Opik 的设计以灵活性为核心。随着数据增长和查询量增加，Opik 与您共同成长。

- **水平扩展** - 增加更多服务副本以即时处理更多流量
- **垂直扩展** - 增加 CPU、内存或存储以处理更密集的工作负载
- **无缝弹性** - 在高峰使用期间扩展，在低谷期间缩减

对于更大的工作负载，ClickHouse 可以扩展以支持企业级部署。常见配置包括：

- 62 个 CPU 核心
- 256 GB RAM
- 25 TB 磁盘空间

ClickHouse 的读取路径也可以通过增加副本进行水平扩展，确保 Opik 在使用量增长时继续保持高性能。

## ClickHouse：高性能存储

Opik 可扩展性的核心是 ClickHouse，这是一款经过验证的高性能分析数据库，专为大规模工作负载设计。Opik 利用 ClickHouse 存储 Trace 和 Span，确保快速查询、稳健的摄入和可靠的性能。

### 实例类型推荐

推荐使用内存优化型实例，内存与 CPU 比例最低为 4:1：

| 部署规模 | 实例规格 |
|------------|----------|
| 小型 | m7i.2xlarge |
| 中型 | m7i.4xlarge |
| 大型 | m7i.8xlarge |

### 复制策略

- **开发环境**：1 个副本
- **生产环境**：2 个副本

为提高效率，应优先进行垂直扩展，再考虑增加副本。

### CPU 与内存指南

目标 CPU 利用率为 10-20%，高峰时可安全达到 40-50%。

保持内存与 CPU 比例至少为 4:1（超大规模环境可扩展至 8:1）。

| 部署类型 | CPU 核心数 | 内存（GiB） |
|------------|-----------|--------------|
| 最低配置 | 2 | 8 |
| 开发环境 | 4 | 16 |
| 生产（小型） | 6 | 24 |
| 生产环境 | 32 | 128 |

### 磁盘建议

为确保在高负载下的可靠性能：

| 配置项 | 推荐值 |
|--------|-------|
| 磁盘类型 | SSD |
| 具体型号 | gp3 |
| 容量 | 8-16 TiB（视工作负载而定） |
| IOPS | 3000 |
| 吞吐量 | 250 MiB/s |

Opik 的 ClickHouse 层在持续大规模摄入下仍能保持弹性，确保查询速度不受影响。

## 摄入最佳实践

数据发送方式比硬件配置更重要。优化摄入模式是提升性能最有效的方法。

### 使用批量端点

Opik 提供批量摄入端点，每请求最多接受 **1,000 项**：

- `POST /v1/private/traces/batch`
- `POST /v1/private/spans/batch`

与其发送 1,000 个单独的 HTTP 请求（每个触发一次数据库插入），单个批量请求可以一次性处理所有数据。这大大减少了连接开销和 ClickHouse 插入压力。

<Tip>
**Opik Python SDK 会自动进行批量处理** — 无需修改代码。本指南主要适用于直接 API 集成或自定义 SDK 实现。
</Tip>

### 启用异步插入

对于生产部署，建议启用 ClickHouse 异步插入。这允许 Opik 缓冲写入并以更大、更高效的批量进行刷新，而不是逐个提交每次插入。

### 设置查询限制

保护您的 ClickHouse 集群免受失控查询的影响：

| 配置项 | 推荐值 |
|---------|-------------------|
| 最大执行时间 | 60 秒 |
| 每次查询最大内存 | 10 GB |
| 每用户最大并发查询数 | 2-4 |

这些限制可以防止单个高开销查询影响摄入性能或其他用户。

### 监控关键指标

健康部署的关键监控指标：

- **ClickHouse CPU 利用率** — 目标平均 10-20%，高峰可安全达到 40-50%
- **插入速率** — 每分钟插入行数，应保持稳定无波动
- **查询持续时间** — p95 和 p99 查询时间；若持续超过 10 秒需排查
- **Pod 重启次数** — 频繁重启可能表明健康检查在负载下超时

## 管理系统表

系统表（如 `system.opentelemetry_span_log`）可能快速增长。为保持存储精简：

- 在 ClickHouse 中配置 TTL 设置，或
- 定期进行手动清理

## Opik 可自信扩展的原因

- **企业就绪** — 支持 TB 级数据量
- **弹性灵活** — 轻松调整资源以匹配工作负载需求
- **稳健可靠** — 专为高可用性和长期稳定性设计
- **面向未来** — 已验证可在无需重新设计的情况下支持增长使用

使用 Opik，您可以从小规模起步并自信扩展，因为您知道您的可观测性平台不会成为瓶颈。
