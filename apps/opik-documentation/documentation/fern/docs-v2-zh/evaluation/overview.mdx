## 为什么要评估您的智能体

LLM 智能体在生产环境中会以无法预测的方式失败。一个对 90% 查询有效的提示词可能会在边缘情况下产生幻觉、忽略上下文，或在用户期望简洁回答时生成冗长的回复。人工审查无法规模化，而且您无法在上线前预见所有失败模式。

您需要自动化回归测试——但不是那种从头开始编写测试套件的方式。最有效的测试套件是增量构建的，来自真实的生产故障。每次发现错误响应时，您将其转化为测试用例。随着时间推移，您的套件将成为针对智能体实际遇到的特定失败模式的全面防护。

<Tip>
  测试套件是在您调试和改进智能体的过程中创建的——它们从真实故障中有机增长，而非来自单独的测试编写阶段。
</Tip>

## 评估循环

<Steps>

### 在生产环境中发现问题

从 Opik 仪表板开始。浏览追踪数据，按错误状态或低反馈评分筛选，点击进入追踪查看完整的跨度树——每个 LLM 调用、工具调用和检索步骤的输入、输出和延迟。

### 添加到测试套件

将失败转化为测试用例。使用自然语言断言将追踪添加到测试套件，该断言描述了预期行为——例如，*"响应不得对上下文中不存在的事实产生幻觉"*。您可以通过 [Ollie](/tracing/debug-agents)（Opik 的 AI 助手）、UI 或 SDK 来完成此操作。

### 更新您的智能体

修复根本原因。通过[提示词库](/development/prompt-library/getting-started)更新提示词，调整工具定义，或更改检索参数。使用 Ollie 帮助诊断问题并提供修复建议。

### 使用测试套件验证

对更新后的智能体运行测试套件。套件会检查每个测试用例——包括新增的——以确认修复有效且没有引入新的回归问题。

</Steps>

每个循环都会添加新的测试用例。随着时间推移，您的测试套件将成为针对智能体真实失败模式的全面回归防护。

## 两种评估方法

Opik 提供两种互补的评估方法：

- **测试套件**：定义自然语言断言，让 LLM 自动检查。最适合对特定行为进行通过/失败测试。
- **数据集和指标**：使用预构建或自定义指标对数据集评分评估智能体输出。最适合通过定量评分衡量多个追踪的质量。

## 核心功能

- **测试套件**：支持自然语言断言和执行策略
- **30+ 预构建指标**：涵盖幻觉、相关性、连贯性等
- **自定义指标**：用于特定领域的评估
- **实验追踪**：并排比较不同版本
- **标注队列**：支持人在回路审查
